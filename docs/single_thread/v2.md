从 flamegraph 看，最大优化空间很明确：

1. **别用 `read_line` / `String`**

   * `read_line` 约 **28%**
   * `from_utf8 / UTF-8 validation` 约 **17%**
   * `to_string / malloc` 约 **5.4%**

   1BRC 输入是 ASCII，可以改成：

   ```rust
   let bytes = std::fs::read(path)?;
   for line in bytes.split(|&b| b == b'\n') {
       // 直接处理 &[u8]
   }
   ```

   或者 `BufRead::fill_buf` 手写扫描，避免每行构造 `String` 和 UTF-8 校验。

2. **别用 `HashMap<String, ...>` 的默认 hasher**

   * `HashMap::entry/find` 约 **46%**
   * `make_hash` 约 **16%**
   * `SipHash finish/write` 很重

   默认 `RandomState/SipHash` 太慢。换成：

   ```rust
   use rustc_hash::FxHashMap;
   ```

   或 `ahash::AHashMap`。1BRC station 名称固定且数量很少，hash 成本占比会非常明显。

3. **避免每次查表都比较 `String`**

   * `eq / memcmp` 约 **5%**

   可以把 key 改成 `&[u8]` / 自定义 station intern：

   * 第一次遇到站名时复制一份；
   * 后续用 hash 定位；
   * 或者自己实现固定容量 hash table，key 存 offset/len。

4. **手写解析温度**
   现在 `parse_temperature` 约 **8.4%**，还有优化空间。1BRC 温度格式固定，大概是：

   ```text
   -12.3
   4.5
   23.4
   ```

   可以直接按 byte 解析成 `i64`，不要 `parse::<f64>()`，也不要 split 成 `&str`。

   示例思路：

   ```rust
   fn parse_temp(b: &[u8]) -> i64 {
       let mut i = 0;
       let neg = b[0] == b'-';
       if neg { i += 1; }

       let mut n = 0i64;
       while b[i] != b'.' {
           n = n * 10 + (b[i] - b'0') as i64;
           i += 1;
       }

       let v = n * 10 + (b[i + 1] - b'0') as i64;
       if neg { -v } else { v }
   }
   ```

5. **避免 `split_once<char>`**
   你现在还有 `split_once<char>` 热点。因为分隔符固定是 `;`，直接在 bytes 里找：

   ```rust
   let semi = memchr::memchr(b';', line).unwrap();
   let name = &line[..semi];
   let temp = &line[semi + 1..];
   ```

   如果已经整体手写扫描，就连 `memchr` 都可以省掉：边扫边找 `;` 和 `\n`。

优先级我建议是：

**`read_line/String` → `bytes` 扫描**，然后 **换 hasher**，再 **手写温度解析**。
这三个改完，单线程应该会快很多。
