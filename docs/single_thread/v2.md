从 flamegraph 看，最大优化空间很明确：

1. **别用 `read_line` / `String`**

   * `read_line` 约 **28%**
   * `from_utf8 / UTF-8 validation` 约 **17%**
   * `to_string / malloc` 约 **5.4%**

   1BRC 输入是 ASCII，可以改成：

   ```rust
   let bytes = std::fs::read(path)?;
   for line in bytes.split(|&b| b == b'\n') {
       // 直接处理 &[u8]
   }
   ```

   或者 `BufRead::fill_buf` 手写扫描，避免每行构造 `String` 和 UTF-8 校验。

2. **别用 `HashMap<String, ...>` 的默认 hasher**

   * `HashMap::entry/find` 约 **46%**
   * `make_hash` 约 **16%**
   * `SipHash finish/write` 很重

   默认 `RandomState/SipHash` 太慢。换成：

   ```rust
   use rustc_hash::FxHashMap;
   ```

   或 `ahash::AHashMap`。1BRC station 名称固定且数量很少，hash 成本占比会非常明显。

3. **避免每次查表都比较 `String`**

   * `eq / memcmp` 约 **5%**

   可以把 key 改成 `&[u8]` / 自定义 station intern：

   * 第一次遇到站名时复制一份；
   * 后续用 hash 定位；
   * 或者自己实现固定容量 hash table，key 存 offset/len。

4. **手写解析温度**
   现在 `parse_temperature` 约 **8.4%**，还有优化空间。1BRC 温度格式固定，大概是：

   ```text
   -12.3
   4.5
   23.4
   ```

   可以直接按 byte 解析成 `i64`，不要 `parse::<f64>()`，也不要 split 成 `&str`。

   示例思路：

   ```rust
   fn parse_temp(b: &[u8]) -> i64 {
       let mut i = 0;
       let neg = b[0] == b'-';
       if neg { i += 1; }

       let mut n = 0i64;
       while b[i] != b'.' {
           n = n * 10 + (b[i] - b'0') as i64;
           i += 1;
       }

       let v = n * 10 + (b[i + 1] - b'0') as i64;
       if neg { -v } else { v }
   }
   ```

5. **避免 `split_once<char>`**
   你现在还有 `split_once<char>` 热点。因为分隔符固定是 `;`，直接在 bytes 里找：

   ```rust
   let semi = memchr::memchr(b';', line).unwrap();
   let name = &line[..semi];
   let temp = &line[semi + 1..];
   ```

   如果已经整体手写扫描，就连 `memchr` 都可以省掉：边扫边找 `;` 和 `\n`。

优先级我建议是：

**`read_line/String` → `bytes` 扫描**，然后 **换 hasher**，再 **手写温度解析**。
这三个改完，单线程应该会快很多。

--- 
FixedMap (v3) vs FxHashMap — 性能分析

### 火焰图数据对比

我把当前 `v3` (FixedMap) 火焰图的关键热点整理如下：

| 热点函数 | 采样数 | 占比 |
|----------|--------|------|
| `read_file_by_bytes` | 121.5M | 96.17% |
| `process_line` / `process_line_inner` | 93.4M | 73.93% |
| `update` (FixedMap) | 55.4M | **43.88%** |
| `update_with_hash` (线性探测) | 47.9M | **~38%** |
| `__memcmp_avx2_movbe` (key 比较) | 18.1M | **~14.36%** |
| `parse_temperature_by_bytes` | 12.0M | 9.47% |
| `fast_hash` | 11.8M | 9.36% |

**核心发现：FixedMap 的查找逻辑（`update_with_hash` + `__memcmp`）占了总时间的 ~38%，是整个程序最大的瓶颈。**

---

### 为什么不如 FxHashMap？

#### 1. Entry 结构体太大，缓存效率极差（根本原因）

你的 `Entry` 结构体：

```rust
struct Entry {
    hash: u64,          // 8 bytes
    key: [u8; 32],      // 32 bytes
    len: u8,            // 1 byte
    agg: I64Aggregator, // 32 bytes (4 × i64)
    used: bool,         // 1 byte
}
// 对齐后约 80 bytes，cap=16384 → Vec ~1.3 MB
```

在 `update_with_hash` 的线性探测中，每探测一个槽位：

```rust
let e = &mut self.entries[idx];  // 加载整个 80-byte Entry 到缓存行
if !e.used { ... }                // 实际只用了 1 byte
if e.hash == hash ...             // 用了 8 bytes
if e.len as usize == key.len()    // 用了 1 byte
&e.key[..e.len as usize] == key   // 用了最多 32 bytes，还要调 memcmp
```

每次探测都加载 **~80 bytes** 的缓存行，但只需要 **~10 bytes** 做决策。这是一场缓存带宽灾难。

#### 2. FxHashMap 用 Swiss Table 设计，只需 1 byte/探测

FxHashMap 底层是 `hashbrown::HashMap`，它把控制信息抽到一个独立的 **control byte 数组**（1 byte/slot）：

| | FixedMap | FxHashMap (hashbrown) |
|---|---|---|
| 每槽元数据大小 | 80 bytes | **1 byte** (control byte) |
| 每次探测加载 | 80 bytes | 1 byte |
| 缓存行可覆盖槽位数 | ~0.8 个 | **64 个** |
| hash 匹配检查 | 加载后才检查 | SIMD 一次查 16 个槽 |
| 全 key 比较触发条件 | 每步都做 hash+len 检查后比较 | 仅 control byte 匹配时才做 |

**每步探测 FxHashMap 只加载 1 byte，FixedMap 加载 80 bytes —— 差了 80 倍的内存带宽。**

#### 3. `__memcmp_avx2_movbe` 占了 14.36%

即使你在 `update_with_hash` 里做了 hash 和长度前置检查，slice 比较 `&e.key[..e.len as usize] == key` 仍然：
- 需要构造临时 slice（额外开销）
- 调用 `__memcmp_avx2_movbe` 做全 key 逐字节比较

在 FxHashMap 中，control byte 的高 7 位存储了 `hash >> 57`，绝大多数 hash 不匹配在扫描 control byte 阶段就被过滤掉了，根本不会触发全 key 比较。

#### 4. 次要问题：hash 计算做了两趟扫描

```rust
fn process_line_inner(line: &[u8], map: &mut FixedMap) {
    // 第一趟：找分号位置
    let mut semi = 0;
    while line[semi] != b';' { semi += 1; }
    let station = &line[..semi];
    let temp = &line[semi + 1..];
    let val = parse_temperature_by_bytes(temp);
    map.update(station, val);  // 内部又调用 fast_hash → 第二趟扫描 station 字节
}
```

你已经有 `process_line_and_compute_hash`（第 136 行）可以在找分号的同时算 hash，避免了第二趟扫描，但被注释掉了。

---

### 总结

| 原因 | 影响 |
|------|------|
| Entry 80 bytes，探测每次全量加载 | **最大瓶颈**，~38% 时间花在 `update_with_hash` |
| 无 control byte，每次探测都做全 key 比较 | `__memcmp` 占 ~14% |
| FxHashMap SIMD 扫描 control byte | 一次查 16 槽，FixedMap 一次只能查 1 槽 |
| hash 计算与分号扫描分离 | 多一趟内存扫描 |

**改进方向建议：**

1. **直接用 FxHashMap 就好** —— `hashbrown` 的实现已经高度优化
2. 如果坚持自建，把 `Entry` 拆成两个数组：`control_bytes: Vec<u8>` + `entries: Vec<(u64, [u8; 32], u8, I64Aggregator)>`，让探测只扫描 1-byte 控制数组
3. 启用 `process_line_and_compute_hash`，边找分号边算 hash
4. 把 `[u8; 32]` 缩短到实际最大站名长度（最长 ~29 字节），减小 Entry
