pub fn __main() {
    let file = File::open(crate::FILE).unwrap();
    let reader = BufReader::new(file);
    let mut map: HashMap<String, Aggregator> = HashMap::new();

    for line in reader.lines() {
        let line = line.unwrap(); // String，不包含末尾换行符
        let Some((station, val)) = line
            .split_once(';')
            .and_then(|(s, v)| v.parse::<f64>().ok().map(|val| (s, val)))
        else {
            continue;
        };
        map.entry(station.to_string())
            .and_modify(|agg| agg.update(val))
            .or_insert_with(|| Aggregator::new(val));
    }

    let mut entries = map.into_iter().collect::<Vec<_>>();
    entries.sort_by(|(s1, _), (s2, _)| s1.cmp(s2));

    print!("{{");
    for (i, (s, agg)) in entries.iter().enumerate() {
        if i > 0 {
            println!(",");
        }
        print!("{}={:.1}/{:.1}/{:.1}", s, agg.min(), agg.mean(), agg.max());
    }
    println!("}}");
}

后续优化建议按优先级看：

1. **输出格式先修正**
   [src/single_thread.rs](/home/maji/Projects/1brc/src/single_thread.rs:33) 这里用的是 `println!(",")`，会在每个元素之间换行。1BRC 通常期望 `{a=... , b=...}` 这种单行格式，应该考虑用 `print!(", ")`。这不是性能问题，是结果格式问题。

2. **避免 `reader.lines()` 每行分配 `String`**
   [src/single_thread.rs](/home/maji/Projects/1brc/src/single_thread.rs:14) 的 `lines()` 每一行都会创建新的 `String`。大文件下这是主要开销之一。
   后续可以学习两条路线：
   - 用 `read_line(&mut String)` 复用同一个 buffer。
   - 更进一步，读 bytes buffer，用 `\n` 手动切行。

3. **减少 station 名称的重复分配**
   [src/single_thread.rs](/home/maji/Projects/1brc/src/single_thread.rs:22) 每次都 `station.to_string()`，即使这个站点已经在 map 里，也会先构造一个新 `String` 给 `entry` 使用。
   可以后续研究 `HashMap::get_mut` 先查已有 key，只有新站点才分配 `String`。1BRC 站点数量少、行数极大，这个优化很有意义。

4. **温度不要用通用 `f64::parse`**
   [src/single_thread.rs](/home/maji/Projects/1brc/src/single_thread.rs:18) 的 `parse::<f64>()` 很通用，也比较重。1BRC 温度格式固定，通常是一位小数，比如 `-12.3`。
   可以把温度解析成整数十分之一度：`-12.3 -> -123`。这样 `Aggregator` 也可以存 `i64` 的 `min/max/sum/cnt`，最后输出时再格式化成一位小数。对应结构在 [src/lib.rs](/home/maji/Projects/1brc/src/lib.rs:10)。

5. **`Aggregator` 用整数更稳定也更快**
   当前 [src/lib.rs](/home/maji/Projects/1brc/src/lib.rs:11) 用 `f64` 存 `min/max/sum`。对 1BRC 来说，输入只有一位小数，整数十分之一度足够表达全部信息。
   好处：
   - 避免浮点累加误差。
   - 比 `f64` parse 更快。
   - 平均值四舍五入规则更可控。

6. **HashMap 容量可以预估**
   [src/single_thread.rs](/home/maji/Projects/1brc/src/single_thread.rs:12) 现在是默认 `HashMap::new()`。1BRC 站点数量有限，可以用 `HashMap::with_capacity(...)`，减少扩容。收益不如前面几项大，但简单。

7. **排序不是主要瓶颈**
   [src/single_thread.rs](/home/maji/Projects/1brc/src/single_thread.rs:27) 最后 `collect + sort_by` 是合理的。因为 station 数量远小于行数，排序成本很小。这里不用急着优化。

8. **单线程优化到一定程度后再考虑多线程**
   你现在这个文件适合先练单线程性能：IO、解析、分配、HashMap。等单线程版本足够快，再切到多线程分块处理，否则多线程会把问题复杂化，反而不容易判断瓶颈。

我建议你的学习顺序是：先修输出格式，然后把 `lines()` 改成复用 buffer，再优化 `station.to_string()`，最后做整数温度解析。这条路线每一步都能单独 benchmark，容易看到效果。
