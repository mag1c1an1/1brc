# jon

这个文件基本就是一个高排名 1BRC 解法，里面几乎每个地方都在为 CPU 服务。

如果从优化层次分析，可以分成下面几层：

---

# 第一层：算法层

## mmap

```rust
let map = mmap(&f);
```

直接把整个文件映射成：

```rust
&[u8]
```

避免：

```text
read
copy kernel -> user
fill_buf
consume
leftover
```

后面所有线程直接访问同一个 mmap。

---

## 分块

```rust
chunk_size = map.len() / nthreads;
```

然后：

```rust
find_newline(&map[end..])
```

找到下一个换行。

这样保证：

```text
thread1: [0.............]
thread2: [..............]
thread3: [..............]
```

没有跨行。

这是标准 1BRC 并行方案。

---

# 第二层：数据结构层

最有意思的是这个：

```rust
HashMap<StrVec, Stat, FastHasherBuilder>
```

而不是：

```rust
HashMap<String, Stat>
```

---

## StrVec

这个相当于 SSO

类似：

```cpp
std::string
small string optimization
```

---

短字符串：

```rust
union StrVec {
    inlined: [u8; INLINE],
}
```

直接塞结构体里。

station 名一般：

```text
Berlin
Tokyo
Paris
```

都很短。

所以：

```text
无 malloc
无 free
无额外指针跳转
```

---

长字符串才：

```rust
Box<[u8]>
```

堆分配。

---

你自己的实现：

```rust
Entry {
    key: [u8; 100]
}
```

其实和这个思路很像。

甚至你的更激进。

---

# 第三层：Hash

这是最骚的部分。

---

他没有用：

```rust
FxHash
AHash
SipHash
```

---

而是：

```rust
FastHasher
```

只 hash 前几个字节。

看这里：

```rust
0..4 => {
    lo
    mid
    hi
}
```

或者：

```rust
acc ^= first_u32
```

---

换句话说：

```text
Berlin
Berlin2
Berlin3
```

可能 hash 冲突。

但没关系。

因为：

```text
1BRC station 数量只有 413
```

所以：

```text
hash质量不重要
hash速度最重要
```

---

这个思想特别重要：

他不是在做通用 HashMap。

他是在做：

```text
专门给 413 个 station 用的 HashMap
```

# 第五层：解析

## split_at_semicolon

最牛的地方。

一般人：

```rust

```

他：


因为：



所以：

```text
';'
一定在最后 4~6 字节附近
```

---

于是：

```text
从尾巴找
```

比从头扫快。
